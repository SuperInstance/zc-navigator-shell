### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mounts.
- Don't rely on file extensions alone; READMEs can be `.md`, `.txt`, `.rst`, or have no extension.
- Don't modify files without reading them first; some may be symlinks or have special permissions.
- Don't ignore hidden directories (`.git`, `.github`) which may contain CONTRIBUTING or other docs.
- Don't assume English; some READMEs may be in other languages.
- Don't evaluate based solely on length; a short, clear README can be good.
- Don't skip repos that appear empty; they might have a README in a subdirectory or branch.
- Don't trust file timestamps alone; a recently updated README might still be poor quality.

### Step 2 — Identify safe channels (P1)
**Safe channels:**
- Use `find` command with appropriate paths (`/home`, `/mnt`, `/opt`) to locate repos.
- Check for README files in root of each repo.
- Read each README fully before scoring.
- Score based on objective criteria: presence, clarity, structure, examples, links.
- Prioritize repos that are actively maintained (recent commits).
- Focus on SuperInstance-related repos (look for `cocapn`, `superinstance`, `plato`, `fleet` in paths).
- Use a simple scoring system (0-10) to rank.

### Step 3 — Optimize within channel (P2)
**Execution plan:**
1. Locate SuperInstance repos across known directories.
2. For each repo, check for README in root.
3. Score each README on:
   - **Existence (0/2):** 0 if missing, 1 if placeholder, 2 if present.
   - **Clarity (0/2):** 0 if confusing, 1 if basic, 2 if clear purpose and usage.
   - **Structure (0/2):** 0 if wall of text, 1 if sections, 2 if well-organized with headers.
   - **Examples (0/2):** 0 if none, 1 if minimal, 2 if runnable code snippets.
   - **Links (0/2):** 0 if no links, 1 if internal, 2 if links to docs/related repos.
4. Sum scores (max 10) and sort ascending.
5. List top 10 worst READMEs with scores and brief rationale.
