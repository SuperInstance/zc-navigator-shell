# Cycle 138 — Navigator
**Task:** Scan SuperInstance repos for READMEs that need improvement. List the 10 worst ones.

---

### Step 1 — Map the Negative Space (P0)
**Rocks (what NOT to do):**
- Don't assume all repos are under `/home` — some may be in other directories or mounted volumes.
- Don't rely on file extensions alone (e.g., `README`, `README.md`, `README.txt`).
- Don't evaluate READMEs without reading their actual content — file size or presence isn't enough.
- Don't ignore nested READMEs (e.g., in `docs/`, `src/` subdirectories).
- Don't process binary files or non-text files mistakenly named README.
- Don't make subjective "worst" calls without defined criteria.
- Don't skip repos that are part of the core fleet (plato-torch, holodeck-rust, etc.).
- Don't assume the task is only about SuperInstance-branded repos — "fleet" may include all related projects.
- Don't fail to check for broken links, outdated examples, or missing setup steps.
- Don't produce a list without actionable improvement points.

---

### Step 2 — Identify Safe Channels (P1)
**Safe channels (what I CAN do):**
- Use `find` command to locate README files recursively from likely root directories (e.g., `/home`, `/fleet`).
- Filter for readable text files using `file` command or extension patterns.
- Read each README fully to assess quality against a standard rubric.
- Define objective criteria for "needs improvement": missing sections, outdated info, poor formatting, lack of examples, no contact/contributing info, etc.
- Prioritize repos that are actively maintained or critical to fleet operations.
- Limit to the top 10 based on severity of issues.
- Provide specific, constructive feedback for each listed README.
- Cross-reference with repo activity (git log) to avoid flagging archived/experimental projects unfairly.
- Output in a structured format: repo name, README path, issues, suggested fixes.

---

### Step 3 — Optimize Within Channel (P2)
**Execution plan:**
1. **Locate READMEs:**  
   ```bash
   find /home /fleet -type f -iname "README*" -not -path "*/.git/*" 2>/dev/null | head -50
   ```
   (Start with 50 candidates to manage scope.)

2. **Assess each README:**  
   Criteria checklist:
   - [ ] Has a clear project description
   - [ ] Includes installation/usage instructions
   - [ ] Lists dependencies or requirements
   - [ ] Provides examples or demos
   - [ ] Documents API or key functions
   - [ ] Links to contributing guidelines
   - [ ] Shows recent updates (within last year)
   - [ ] Formatting is consistent (Markdown, headings, code blocks)
   - [ ] No obvious typos or broken links

3. **Score and rank:**  
   Assign penalty points for each missing criterion. Higher penalty = more urgent need for improvement.

4. **Select top 10:**  
   Take highest penalty scores, ensuring diversity across fleet domains (not all from one project).

5. **Write output:**  
   Present as a table with repo, README path, penalty score, and specific issues.
