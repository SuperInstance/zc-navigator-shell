**Task:** Scan SuperInstance repos for READMEs that need improvement. List the 10 worst ones.

---

### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mounts.
- Don't rely on file extensions alone; READMEs can be `.md`, `.txt`, or have no extension.
- Don't modify files without reading them first — understand the context.
- Don't rank based solely on length; a short, clear README can be good.
- Don't ignore hidden directories (`.git`) or nested repos.
- Don't assume the repo root is obvious; look for `.git/config` or `Cargo.toml`/`package.json`.
- Don't process binary files or non-text files as READMEs.
- Don't skip repos that have no README — they are candidates for creation, not improvement.

### Step 2 — Identify safe channels (P1)
**Safe channels:**
- Use `find` command with `-type f` and case-insensitive name matching.
- Limit search to likely directories: `/home`, `/mnt`, `/opt`, `/root`.
- Exclude binary files with `file --mime-type` or `grep -I`.
- Read each README fully to assess structure, clarity, completeness.
- Define scoring criteria: presence, structure, examples, links, freshness.
- Output a ranked list with specific deficiencies.

### Step 3 — Optimize within channel (P2)
**Execution plan:**
1. Find all README files in fleet directories.
2. Read and score each based on defined criteria.
3. Sort by score (lowest = worst).
4. Output top 10 with repo path and deficiencies.

### 2. Score each README
**Scoring criteria (0-10, 10=best):**
- **Presence (2 pts):** File exists? (2 if yes, 0 if no)
- **Structure (3 pts):** Has clear sections (Overview, Usage, Examples, API)? (0-3)
- **Examples (2 pts):** Includes code snippets or usage examples? (0-2)
- **Links (2 pts):** Links to related repos, docs, issues? (0-2)
- **Freshness (1 pt):** Updated in last year? (1 if yes, 0 if no)
