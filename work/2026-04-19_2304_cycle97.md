### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mount points.
- Don't rely on file extensions alone; READMEs can be `.md`, `.txt`, `.rst`, or have no extension.
- Don't evaluate repos I cannot read due to permissions.
- Don't waste cycles on temporary or archived repos (e.g., `*-old`, `*-backup`).
- Don't hallucinate content; base scoring on actual file inspection.

### Step 2 — Identify safe channels (P1)
**Safe Paths:**
- Use `find` command to locate all `README*` files within known fleet directories.
- Filter to repos under `/home` and `/mnt` (common locations from prior scans).
- Read each README's first 50 lines to assess quality.
- Apply objective scoring criteria: presence of key sections, clarity, length.
- Sort by score and list the 10 lowest.

### Step 3 — Optimize within channel (P2)
**Execution:**
1. **Locate READMEs:**
   ```bash
   find /home /mnt -type f -name "README*" -o -name "readme*" 2>/dev/null | head -100
   ```
   *(Note: Actual run limited to 100 results for efficiency; fleet has many repos.)*

2. **Score each README (0-10 scale):**
   - **+2 points** if it has a clear project description.
   - **+2 points** if it has installation/usage instructions.
   - **+2 points** if it has a dependency list or environment setup.
   - **+2 points** if it links to related repos or documentation.
   - **+2 points** if it is longer than 10 lines of substantive content.
   - **-5 points** if it is empty or contains only placeholders (e.g., "TODO").

3. **Rank and list the 10 worst.**
