# Cycle 51 — Navigator
**Task:** Scan SuperInstance repos for READMEs that need improvement. List the 10 worst ones.

---

### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mounts.
- Don't rely on file extensions alone; READMEs may be `.md`, `.txt`, `.rst`, or have no extension.
- Don't modify files during scan; this is an analysis phase only.
- Don't include repos that are intentionally minimal (e.g., placeholder, archived).
- Don't trust `ls` output alone; need to check actual file content.

### Step 2 — Identify safe channels (P1)
**Safe paths:**
- Use `find` command to locate README files recursively from known fleet directories.
- Check file size and content length to filter empty or near‑empty READMEs.
- Use a simple scoring heuristic: smaller size + less structure = worse README.
- Focus on active repos (recent commits, not archived).
- Validate by reading the first 10 lines of each candidate.

### Step 3 — Optimize within channel (P2)
**Execution:**
1. Locate fleet repositories.
2. For each repo, find README file.
3. Score each README based on:
   - File size (bytes)
   - Presence of key sections (Description, Usage, Installation, Examples)
   - Whether it’s a placeholder (e.g., “TODO”, “Coming soon”)
4. Rank worst 10 by score.
5. Output list with repo path and reason.

## Scan Results

**Base directory used:** `/home` (recursive search from here)

**Command used:**
```bash
find /home -type f -iname "README*" -path "*/.git/*" -prune -o -type f -iname "README*" -print 2>/dev/null | head -100
```

**Scoring method:**
- Size < 200 bytes → +5 points (worse)
- No Markdown headings → +3 points
- Contains “TODO” or “Coming soon” → +2 points
- No code block or example → +2 points
- Single paragraph only → +3 points
Higher score = worse README.
