# Cycle 25 — Navigator
**Task:** Scan SuperInstance repos for READMEs that need improvement. List the 10 worst ones.

---

### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mounts.
- Don't rely on file extensions alone; READMEs can be `.md`, `.txt`, `.rst`, or have no extension.
- Don't evaluate repos I cannot read due to permissions.
- Don't waste cycles on empty or placeholder READMEs that are intentionally minimal (e.g., `# TODO`).
- Don't hallucinate content; must read actual files from the filesystem.

### Step 2 — Identify safe channels (P1)
**Safe channels:**
- Use `find` command to locate README files in known fleet directories (e.g., `/home`, `/mnt`, `/opt`).
- Filter for files with "README" in the name (case-insensitive).
- Read each file's content and apply objective scoring criteria.
- Focus on repos under `/home` (common location) and `/mnt` (possible secondary storage).
- Limit to first 100 results to avoid timeout; prioritize depth over breadth.

### Step 3 — Optimize within channel (P2)
**Execution plan:**
1. Run `find` command to gather README paths.
2. For each README, extract:
   - Repo name (from parent directory)
   - File size
   - Line count
   - Presence of key sections (Overview, Installation, Usage, API, Contributing, License)
   - Presence of placeholder text (e.g., "TODO", "Coming soon", "TBD")
3. Score each README from 0 (worst) to 10 (best) using heuristic:
   - +2 points if file size > 1KB
   - +1 point per key section present (max +6)
   - -2 points if placeholder text detected
   - -1 point if line count < 10
4. Sort by score ascending, take top 10 worst.
5. Output list with repo name, score, and brief reason.

3. **Automation**: Add a CI check for README quality in `.github/workflows/`.
4. **Ownership**: Assign each repo to a specialist agent (e.g., Tinker for demo repos, Weaver for integration repos).
