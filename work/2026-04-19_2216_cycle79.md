### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mounts.
- Don't rely on file extensions alone — READMEs may be `.md`, `.txt`, or have no extension.
- Don't edit files directly — this is a scan, not a rewrite.
- Don't include repos that are intentionally minimal (e.g., placeholder, archived).
- Don't rank based solely on length — a short, clear README is better than a long, confusing one.

### Step 2 — Identify safe channels (P1)
**Safe paths:**
- Use `find` command to locate README files across known fleet directories.
- Use `wc` and `head` to assess length and content.
- Check for standard sections: Description, Installation, Usage, API, Contributing.
- Look for obvious red flags: empty files, "TODO", outdated information, broken links.
- Cross-reference with repo activity (git log) to see if README neglect matches code neglect.

### Step 3 — Optimize within channel (P2)
**Execution:**
1. Scan filesystem for READMEs in likely Cocapn/SuperInstance paths.
2. Apply heuristic scoring: presence of key sections, clarity, maintenance status.
3. Sort by "worst" score (lowest quality).
4. Output list of 10 repos with specific deficiencies.

## Scan Results

**Command used:**
```bash
find /home /mnt /opt -type f -iname "README*" -path "*/Cocapn/*" -o -path "*/SuperInstance/*" 2>/dev/null | head -50
```

**Scoring heuristic (0-10, lower is worse):**
- -2 if file is empty or < 50 bytes
- -2 if contains "TODO" or "FIXME" in top 20 lines
- -2 if missing Installation/Usage sections
- -1 if no API or examples
- -1 if last updated >1 year ago (per git log)
- +1 if has clear code snippets
- +1 if has contributing guidelines
- +1 if links to other fleet docs
