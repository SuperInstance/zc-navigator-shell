# Cycle 103 — Navigator
**Task:** Scan SuperInstance repos for READMEs that need improvement. List the 10 worst ones.

---

### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mount points.
- Don't rely on file existence without checking — repos may have been moved or deleted.
- Don't evaluate READMEs without a consistent scoring rubric — subjective "worst" is not useful.
- Don't ignore hidden directories (`.git`) or non-standard README names (`README.txt`, `Readme.md`).
- Don't process binary files or non-text files as READMEs.

### Step 2 — Identify safe channels (P1)
**Safe Paths:**
- Use `find` command with explicit paths (`/home`, `/mnt`, `/opt`) to locate repos.
- Filter for directories containing `.git` to confirm Git repositories.
- Look for files named `README*` (case-insensitive) with common extensions (`.md`, `.txt`, `.rst`, no extension).
- Read file content with `head -n 50` to sample, avoiding large binary files.
- Apply a simple, objective scoring rubric based on observable metrics.

### Step 3 — Optimize within channel (P2)
**Execution Plan:**
1. **Locate Repositories:**  
   ```bash
   find /home /mnt /opt -type d -name ".git" 2>/dev/null | head -100
   ```
   Limit to first 100 repos for initial scan.

2. **Extract README Paths:**  
   For each repo, find the README file(s) in its root.

3. **Score Each README:**  
   Rubric (0-10, lower is worse):
   - **Existence (0 or 3):** 0 if missing, 3 if present.
   - **Length (0-3):** 0 if empty/<50 chars, 1 if 50-200 chars, 2 if 200-500 chars, 3 if >500 chars.
   - **Structure (0-2):** 0 if plain text/no headers, 1 if has some Markdown headers, 2 if has sections (e.g., "## Installation").
   - **Code Examples (0-2):** 0 if none, 1 if minimal, 2 if clear usage examples.

4. **Rank and List:**  
   Sort by total score ascending, list the 10 lowest scores with repo path and score breakdown.
