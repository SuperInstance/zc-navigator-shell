### Step 1 — Map the Negative Space (P0)
**Rocks (what NOT to do):**
- Don't assume all repos are under a single directory path without checking the actual fleet-knowledge/ structure.
- Don't rely on memory or hallucinated repo names; read the actual file list.
- Don't evaluate READMEs without a clear, objective scoring rubric.
- Don't submit a list without providing specific, actionable reasons for each selection.
- Don't ignore the context that READMEs serve different purposes (library vs. tool vs. experiment).

### Step 2 — Identify Safe Channels (P1)
**Safe Channels (where I CAN operate):**
- I can read the `fleet-knowledge/` directory tree to find repository documentation.
- I can use a simple, consistent scoring system based on observable README attributes.
- I can focus on repos that are likely active (e.g., have recent commits, are referenced elsewhere).
- I can provide concrete examples of deficiencies and suggest improvements.
- I can limit the list to 10, as requested, even if more candidates exist.

### Step 3 — Optimize Within Channel (P2)
**Action Plan:**
1. **Scan:** Examine `fleet-knowledge/` for files that appear to be READMEs or primary documentation for a repo/project.
2. **Score:** Apply a rubric with criteria: Existence, Completeness (Purpose, Usage, API), Clarity, and Maintenance (up-to-date links/examples).
3. **Rank:** Sort by lowest score.
4. **List:** Present the 10 worst with scores, reasons, and a brief improvement directive.

**Rubric (Score 0-10, lower is worse):**
- **Existence (0 or 3):** 0=No README/file, 3=File exists.
- **Completeness (0-4):** 0=Missing all, 1=Purpose only, 2=+Usage, 3=+API/Key Functions, 4=All + examples.
- **Clarity (0-2):** 0=Unreadable, 1=Understandable with effort, 2=Clear and well-structured.
- **Maintenance (0-1):** 0=Broken links/outdated code, 1=Functional.
