**Rocks (what NOT to do):**

- Don't rely on file names alone; must read content to assess quality.

- Don't rank based on subjective "bad writing"; need objective criteria.
- Don't process binary files or non-text files as READMEs.

### Step 2 — Identify safe channels (P1)
**Safe channels (where I CAN operate):**
- Scan known fleet directories from `fleet-knowledge/` for repo paths.
- Use `find` command to locate README.md, README, README.txt files.
- Read file content and apply scoring criteria.
- Output list sorted by score (lowest = worst).
- Limit to top 10 worst for this task.

### Step 3 — Optimize within channel (P2)
**Best path:**
1. Use `find` from root (`/`) with `-name` patterns for README files.
2. Filter to likely repo directories (contain `.git` or common source files).
3. Score each README using objective criteria.
4. Sort and output top 10 worst.

**Scoring criteria (0=worst, 10=best):**
- **Existence (0 or 10):** 0 if missing, 10 if present.
- **Length (0-3):** 0=empty, 1=<50 words, 2=50-200 words, 3=>200 words.
- **Structure (0-3):** 0=no headers, 1=some headers, 2=clear sections, 3=full template (Purpose, Usage, etc.).
- **Clarity (0-2):** 0=unclear, 1=understandable, 2=concise and clear.
- **Code examples (0-2):** 0=none, 1=minimal, 2=working examples.
- **Maximum score:** 20.
