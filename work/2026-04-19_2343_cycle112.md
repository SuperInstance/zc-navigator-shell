**Task:** Scan SuperInstance repos for READMEs that need improvement. List the 10 worst ones.

### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mount points.
- Don't rely on file existence without checking — repos may be missing, moved, or empty.
- Don't evaluate READMEs without a consistent scoring rubric — subjective "bad" is not useful.
- Don't ignore hidden directories (`.git`) or non-standard README names (`README.txt`, `Readme.md`).
- Don't process binary files as text — could crash the scan.

### Step 2 — Identify safe channels (P1)
**Safe channels:**
- Use `find` command with explicit paths and file patterns.
- Filter for only text files using `file` command or extension checks.
- Read actual file content to assess quality against objective criteria.
- Limit search to known Cocapn/SuperInstance directories from fleet knowledge.
- Handle errors gracefully — skip unreadable files, continue scanning.

### Step 3 — Optimize within channel (P2)
**Execution plan:**
1. **Locate repos:** Search `/home`, `/mnt`, `/opt` for directories containing `.git` subdirectories.
2. **Find READMEs:** In each repo, look for files matching `README*` (case-insensitive).
3. **Score each README:** Apply rubric:
   - **Existence (0/1):** File exists and is non-empty.
   - **Structure (0-3):** Has clear sections (Description, Installation, Usage, etc.).
   - **Clarity (0-3):** Language is clear, not overly technical or vague.
   - **Completeness (0-3):** Covers key aspects: what, why, how, examples.
   - **Formatting (0-2):** Proper Markdown, code blocks, links.
   - **Total score:** Sum (0-12). Lower score = worse.
4. **Rank:** Sort by total score ascending, pick top 10 worst.
5. **Output:** List with repo path, README path, score, and brief reason.
