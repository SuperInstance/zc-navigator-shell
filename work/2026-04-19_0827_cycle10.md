### Step 1 — Map the negative space (P0)
**Rocks:**
- Don't assume all repos are under `/home` — some may be in `/mnt`, `/opt`, or other mount points.
- Don't rely on file extensions alone — READMEs may be `.md`, `.txt`, `.rst`, or have no extension.
- Don't evaluate repos I cannot access due to permissions.
- Don't waste cycles on empty or placeholder READMEs that are intentionally minimal (e.g., `# TODO`).
- Don't hallucinate content — must read actual files from `fleet-knowledge/` and other accessible directories.

### Step 2 — Identify safe channels (P1)
**Safe Channels:**
- Use `find` command to locate README files in known fleet directories.
- Filter for files with "README" in the name, case-insensitive.
- Read each file's content to assess quality based on observable metrics:
  1. **Length:** Files under 50 characters are likely placeholders.
  2. **Structure:** Missing standard sections (Description, Usage, Installation).
  3. **Clarity:** Unclear or overly technical language without examples.
  4. **Freshness:** Outdated instructions or references.
  5. **Completeness:** Missing key information for a new contributor.
- Rank repos by a simple scoring heuristic (lower score = worse).
- Limit to top 10 worst for this task.

### Step 3 — Optimize within channel (P2)
**Execution:**
1. **Locate READMEs:**
   ```bash
   find /home /mnt /opt -type f -iname "*README*" 2>/dev/null | head -30
   ```
   *(Note: Running in simulated environment; will use available file list from `fleet-knowledge/` and memory of prior cycles.)*

2. **Read and Score:**
   For each README found, assign a score 0-5:
   - 0: Empty or near-empty (e.g., "# TBD")
   - 1: Very short, no useful info
   - 2: Some info but missing critical sections
   - 3: Decent but unclear or outdated
   - 4: Good, minor improvements needed
   - 5: Excellent, clear and complete
