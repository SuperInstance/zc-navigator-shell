### Step 1 — Map the Negative Space (P0)
**Rocks (what NOT to do):**
- Don't assume all repos are under a single directory path without scanning.
- Don't rely on file names alone; must read content to assess quality.
- Don't ignore repos with no README at all (they are candidates).
- Don't rank based on subjective "bad writing"; need objective criteria.
- Don't process binary files or non-text files as READMEs.

### Step 2 — Identify Safe Channels (P1)
**Safe Channels (where I CAN operate):**
- Use `find` command to locate README files (README, README.md, README.txt, etc.).
- Use `wc` and `file` to filter text files and get basic metrics (line count, word count).
- Read file content to check for presence of key sections (Overview, Installation, Usage).
- Score based on objective heuristics: file size, section presence, example presence.
- Limit to first 10 worst scores for final output.

### Step 3 — Optimize Within Channel (P2)
**Optimal Path:**
1. **Scan for repos:** Use `find /fleet-knowledge -type f -name "README*" -o -name "readme*"` to locate files.
2. **Filter and read:** For each file, check if it's text, get line count, and scan for key phrases.
3. **Score each README:** Apply scoring rubric:
   - -10 points if file is empty or < 5 lines.
   - -5 points if missing "Overview" or "Description" section.
   - -5 points if missing "Installation" or "Setup" section.
   - -5 points if missing "Usage" or "Examples" section.
   - -2 points if no code blocks or examples.
   - +1 point per 10 lines (up to +10) for length.
   - Final score = sum of penalties + length bonus (lower is worse).
4. **Sort and list:** Sort by score ascending, take top 10 worst.
5. **Output:** List filenames and scores with brief reason.
