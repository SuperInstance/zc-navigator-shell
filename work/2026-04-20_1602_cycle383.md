### Step 1 — Map the Negative Space (P0)
**Rocks (what NOT to do):**
- Don't assume all repos are under a single directory path without checking the actual file system.
- Don't rely on memory or hallucinated repo names; read from the actual `fleet-knowledge/` index if available.
- Don't evaluate READMEs without a clear, objective scoring rubric.
- Don't list repos that don't exist in the current shell/git context.
- Don't skip verifying file existence before attempting to read.

### Step 2 — Identify Safe Channels (P1)
**Safe Channels (where I CAN operate):**
- I can read the local git repository structure using shell commands (e.g., `find`, `ls`).
- I can read files from `fleet-knowledge/` as provided in the context.
- I can define a simple, objective scoring system based on observable README attributes.
- I can limit analysis to repos actually present in the working directory.
- I can output a factual list based on the files I can actually access.

### Step 3 — Optimize Within Channel (P2)
**Optimal Path:**
1. **Discover Repos:** Use `find` to locate directories containing a `.git` subdirectory or a `README.md` file, starting from the current directory.
2. **Extract READMEs:** For each discovered repo, attempt to read its `README.md`.
3. **Score READMEs:** Apply a scoring rubric (see below).
4. **Rank:** Sort by score (lowest = worst).
5. **List:** Output the 10 lowest-scoring repos with their scores and deficiencies.

**Scoring Rubric (0-10 points):**
- **Existence (0 or 3 points):** 0 if file missing, 3 if present.
- **Length (0-3 points):** 0 if empty or <50 words, 1 if 50-200 words, 2 if 200-500 words, 3 if >500 words.
- **Structure (0-2 points):** 0 if no headers, 1 if some headers, 2 if clear sections (e.g., "Installation", "Usage").
- **Clarity (0-2 points):** 0 if incomprehensible, 1 if understandable with effort, 2 if clear and concise.

**Maximum Score:** 10. Lower scores indicate greater need for improvement.
